# UX Edge Cases

> Intelligence is judged at its edges. A confident, honest, graceful response when data is thin,
> messy, or absent is what separates a real intelligence operating system from a brittle audit
> tool. Each edge case is handled in a way that *protects the user's state* on the journey rather
> than dropping them back to scepticism.

For each case: the situation, the UX response, and the user-state impact (which state it protects
or advances). The unifying principle comes straight from the Brain (20): **honest limits beat
fabricated certainty, and calibrated abstention is a valid, trust-building output.**

---

## 1. User has no data
- **Response:** proceed in fast-judgement mode on answers + general patterns; deliver real cold
  recognition; be explicit that confidence is limited and what data would raise it.
- **State impact:** keeps them moving from Recognition (2) toward Diagnosis (3) without an empty
  "we can't help you" wall. Never block on missing data.

## 2. User only provides a URL
- **Response:** observe the page live; surface what a URL *can* support (discovery, conversion,
  listing/menu signals) at honest confidence; clearly state what it *can't* (true costs, orders,
  profit) and invite that data later.
- **State impact:** advances Diagnosis→Evidence (3→4) on the visible signals; sets honest
  expectations so trust isn't later broken.

## 3. User uploads messy files
- **Response:** parse what's usable, name what isn't, proceed on the usable subset. "I can use
  these columns; these I'll ignore." Never fail hard on imperfect data.
- **State impact:** protects Trust (5) — graceful handling of mess is itself proof of competence.

## 4. User uploads sensitive data
- **Response:** reassure before anxiety peaks; state plainly that sensitive details aren't
  required; use the minimum, explain the use, offer a lower-data path.
- **State impact:** removes the upload-anxiety friction at Diagnosis (3); protects Trust (5).
  (See `Lead_Capture_Experience.md`, privacy below.)

## 5. Unsupported platform
- **Response:** accept the input generically; reason from transferable patterns (which are
  platform-independent, 19); be honest that platform-specific depth isn't available, and still
  give a useful cross-platform read.
- **State impact:** keeps Recognition/Diagnosis alive instead of a dead end; demonstrates that
  Growmerce reasons in patterns, not just platforms.

## 6. Too many products / SKUs
- **Response:** focus on the vital few (hero SKUs / categories) per the operator heuristic (H2);
  summarise the long tail; don't choke trying to analyse everything.
- **State impact:** advances Action (6) with a prioritised view; demonstrates operator judgement
  (knowing what matters) rather than data overload.

## 7. AI has low confidence
- **Response:** say so plainly, in the four-band language ("منخفضة/متوسطة"); frame the finding as
  a possibility to confirm with a specific cheap next step; never inflate. Offer the test that
  would resolve it.
- **State impact:** protects Evidence/Trust (4→5) — honest low confidence *builds* trust here,
  it doesn't lose it.

## 8. Missing competitor information
- **Response:** infer likely competitors and offer them as editable suggestions (accept/reject/
  edit/add); proceed with what's known; mark competitor findings as lower confidence until
  confirmed.
- **State impact:** keeps Diagnosis (3) moving; the suggestion interaction itself reveals
  intelligence.

## 9. User rejects AI suggestions
- **Response:** accept the correction instantly and gracefully; treat it as valuable input
  (human-in-the-loop, 20); re-reason with their correction; never argue or re-push.
- **State impact:** protects Trust (5) — respecting the operator's knowledge deepens it; reveals
  that the system listens.

## 10. User skips steps
- **Response:** allow it everywhere; confidence flexes down, nothing blocks; the report later
  notes what couldn't be assessed and why. Every step is optional.
- **State impact:** removes form-fatigue friction at Diagnosis (3); keeps momentum toward Action.

## 11. Report generation is slow
- **Response:** show an honest working state that narrates real reasoning steps (not a fake
  spinner); deliver partial value early where possible; never a blank wait.
- **State impact:** protects Evidence/Trust (4→5); reinforces that genuine cognition is happening.

## 12. Mobile user leaves mid-way
- **Response:** front-load value so a partial visit still leaves a recognition; keep a one-tap
  WhatsApp present; allow easy resume; capture nothing prematurely.
- **State impact:** preserves whatever state was reached; offers a one-motion jump to Commitment
  (7) for the interested-but-leaving visitor. (See `Mobile_TikTok_Landing_Experience.md`.)

## 13. Privacy concern
- **Response:** plain-language data-use explanation, available before they have to ask; explicit
  permission asks; a path that needs no sensitive data; honour "no" with zero cost (the preview
  stays theirs where possible).
- **State impact:** removes the agency-distrust/privacy friction blocking Commitment (7);
  protects Trust (5).

## 14. No clear pattern found
- **Response:** say so honestly — "we don't see a strong leak in what you've shared; here's what
  we'd look at with more data." Offer the next investigative step. **Never invent a problem to
  justify a sale.**
- **State impact:** this is the ultimate trust test. Refusing to fabricate *builds* Trust (5) more
  than a manufactured finding ever could; advances toward a genuine conversation (7) on honest
  footing. (GUARD / calibrated abstention, 20.)

---

## 15. The unifying rules for all edge cases

1. **Never block on missing/imperfect data** — degrade gracefully to lower confidence.
2. **Never fabricate** — no invented numbers, no manufactured problems (GUARD, 20).
3. **Always be honest about limits** — and pair the limit with the specific data/step that would
   resolve it.
4. **Always protect the user's state** — an edge case should never drop the visitor back to
   scepticism; ideally it deepens trust by handling the mess well.
5. **Always keep a forward path** — one-tap WhatsApp and a clear next step exist even when the
   analysis can't proceed.

---

## 16. User-State Transition (this surface)

| Lens | Edge-case handling |
|---|---|
| **State entering** | any state, when reality (thin/messy/absent data) threatens to break the transition |
| **State leaving** | the same or advanced state — the edge case is absorbed without regression |
| **Friction** | the fear that "this won't work for my situation / they'll fake it" → removed by graceful, honest handling |
| **Trust gained** | competence and integrity under imperfect conditions — often the *strongest* trust signal on the site |
| **Intelligence revealed** | the Brain's honesty and judgement (abstention, prioritisation, listening) |
| **Next action** | always a forward path: more data, a specific test, or WhatsApp |

Handled this way, edge cases stop being failure modes and become some of the most persuasive
proof on the site that Growmerce is a genuine intelligence operating system, not a brittle tool.
