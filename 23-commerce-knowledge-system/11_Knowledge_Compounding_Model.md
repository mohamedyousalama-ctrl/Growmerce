# 11 — Knowledge Compounding Model

> How the Knowledge System grows, strengthens, evolves, and retires knowledge over time. This is the
> engine of Growmerce's improvement: an institution that *understands more every year* because it
> systematically converts experience into validated, transferable knowledge. Compounding is what
> turns a knowledge base into a moat (15).

---

## 1. The compounding thesis

> Most companies' commercial understanding is static or personal — it lives in individuals and resets
> each engagement. Growmerce's understanding **compounds**: every engagement produces knowledge that
> is captured, validated, generalised, and added to a shared institutional corpus, so the company
> reasons better tomorrow than today — and across *all* businesses, because the knowledge is
> transferable.

Two properties make compounding possible (both established earlier):
- **Transferability** — knowledge is organised by commercial reality (02), not industry, so a lesson
  from one engagement improves reasoning everywhere.
- **Explicit, typed objects** — knowledge is captured as typed objects (03) with provenance and
  certainty, so it can be validated and promoted rather than lost.

---

## 2. How new knowledge is added (the generation paths)

Knowledge enters the system through three paths:

### Path A — Earned from outcomes (the primary, proprietary path)
Execution produces outcomes (did the fix work, here, in this context?). Each outcome is a **case
observation** (03). This is the most valuable path because the data is *exclusive* — only Growmerce,
which operates the fixes, sees whether they worked (links to the Supply Chain's execution signal, 22).

```
ENGAGEMENT → recommendation applied → OUTCOME measured → CASE OBSERVATION → (recurs) → LESSON
```

### Path B — Derived from reasoning
The Brain, reasoning over many engagements, surfaces a recurring un-named reality or a previously
implicit relationship. These become candidate knowledge objects (a new pattern, rule, or model) for
validation.

### Path C — Imported from external expertise
Established commercial knowledge (operator experience, credible literature, domain expertise) is
encoded as knowledge objects. This seeds the system and fills gaps, but is always tagged with its
(external) provenance and held to the same validation.

Path A is what makes the moat *compound* (exclusive and self-reinforcing); Paths B and C give it
breadth and a starting base.

---

## 3. How knowledge strengthens (promotion up the hierarchy)

Knowledge gains strength by **recurring across independent contexts and being confirmed by outcomes** —
the induction pipeline of the Objects Framework (03):

```
CASE OBSERVATION  (seen once)
   → LESSON         (a fix worked/failed, here)
   → confirmed across ≥N independent contexts
   → PATTERN / RULE (a reliable recurring reality)
   → explained by a MODEL / PRINCIPLE (we understand *why*)
```

As an object climbs, its **certainty** rises and its **authority in reasoning** grows (a Principle
outranks an Assumption). Promotion is governed by the Quality Framework (12): an object earns a higher
type only with sufficient independent confirmation. Strength is *earned*, never assumed — the same
discipline as the Supply Chain's source tiers (22).

The convergence rule applies here as it does to evidence (22): a lesson confirmed across **independent
contexts** strengthens; the same lesson repeated in correlated situations does not (anti
double-counting at the knowledge level).

---

## 4. How knowledge evolves

Knowledge is not only added; it is *revised*:
- **Refinement** — a rule's scope is narrowed or widened as evidence accrues ("this holds in low-
  review-density categories specifically").
- **Exception discovery** — a case where a rule failed becomes an **Exception** (03), making the rule
  *more* trustworthy by bounding it.
- **Re-explanation** — a better model supersedes a weaker explanation of the same pattern.
- **Benchmark updating** — reference values shift as markets move (a continuous, expected evolution,
  not a failure).

Evolution keeps knowledge *current* — the antidote to the failure mode of stale knowledge (links to
market-shift knowledge, 07, and Failure Modes, 22).

---

## 5. How outdated knowledge is retired

Knowledge can become wrong (a market shifts, a platform changes its mechanics, a tactic stops
working). Retirement is a managed process, not neglect:
- **Drift detection** — a previously reliable object that starts predicting poorly (outcomes
  contradict it) is flagged for review (links to the Brain's drift monitoring, 20).
- **Deprecation** — a flagged object is marked *under review*, then *deprecated* if confirmed
  outdated; it is not silently deleted (provenance and history are preserved for auditability).
- **Versioning** — knowledge is versioned (12), so the system knows *which* understanding produced a
  past conclusion and can reason about why it changed.
- **Replacement** — outdated knowledge is superseded by current knowledge, with the change recorded.

Retiring stale knowledge is as important as adding new knowledge; an institution that only accumulates
eventually reasons from a museum.

---

## 6. How contradictions are handled

When two knowledge objects conflict, they are resolved by rule (mirroring evidence-conflict
resolution, 22), never by averaging:
1. **Hierarchy** — the higher-certainty / more-general object governs (a confirmed Principle outranks
   an Assumption); the weaker is down-weighted.
2. **Scope/Exception** — often the contradiction is apparent, not real: the two objects apply in
   *different contexts*. The resolution is an **Exception** that bounds each to its scope.
3. **Outcome arbitration** — if both are comparable in strength, accumulated outcomes decide which
   predicts reality; the loser is deprecated.
4. **Held tension** — if genuinely unresolved, both are flagged, confidence in the affected domain is
   lowered, and the contradiction is surfaced honestly (never hidden — links to Unknowns, 13).

## 7. The compounding loop (the whole engine)

```
        ┌────────────────────────────────────────────────────────────┐
        ▼                                                            │
  KNOWLEDGE interprets evidence → produces patterns & recommendations │
        │                                                            │
        ▼                                                            │
  EXECUTION applies playbooks → OUTCOMES (exclusive data)            │
        │                                                            │
        ▼                                                            │
  outcomes → case observations → lessons → (validated) → stronger    │
  knowledge; stale knowledge retired; contradictions resolved        │
        │                                                            │
        └─────────  the system understands more next time  ──────────┘
```

Each pass spends knowledge (to interpret) and earns knowledge (from outcomes), net-positive. Because
the knowledge is transferable, gains made on one engagement raise the floor for *all* future
engagements. This is compounding in the strict sense: the rate of improvement is proportional to the
accumulated base.

---

## 8. Why compounding is structural, not aspirational

Compounding here is not a slogan; it is a consequence of three concrete design choices:
1. **Outcomes are captured as typed knowledge** (not lost) — so experience accumulates.
2. **Knowledge is transferable** — so accumulation helps everywhere, not in one silo.
3. **Promotion/retirement is validated** — so the base gets *truer*, not just bigger.

Remove any one and compounding stops. With all three, Growmerce's understanding grows with every
engagement — and the gap to anyone without this engine widens over time. (The strategic consequence is
the subject of `15_Why_Growmerce_Knows_More.md`.)
